Key Architectural Decisions

Single-page HTML, no framework

Every client site is a single HTML file with inline CSS and JS. No React, no Next.js, no build step. Vercel serves it in milliseconds. Zero cold starts. Zero dependency drift. The client's site works forever with no maintenance.

Claude Haiku for comment replies

The SPARK listener uses Claude Haiku, not Sonnet. Haiku is 20× cheaper and fast enough for real-time webhook responses. Three-sentence replies don't need a £0.015/1k token model. Haiku handles it at £0.00075/1k tokens. At scale, this is the difference between a profitable and unprofitable product.

Promise.allSettled in the publisher

The social publishing layer dispatches to all platforms in parallel using Promise.allSettled — not Promise.all. If TikTok's scope approval is pending, Instagram and LinkedIn still post. One platform failing never blocks the others.

AES-256-GCM token encryption

Every client's social platform OAuth token is encrypted before storage. The encryption key lives in Vercel env vars only — never in the database. A Supabase breach doesn't expose client tokens. The key and the ciphertext are never in the same place.

Always HTTP 200 from webhooks

Both the PayPal webhook and the SPARK listener always return HTTP 200, even on internal errors. Errors are logged internally. Returning non-200 causes PayPal and social platforms to retry indefinitely, creating queue bloat and duplicate actions. The webhook confirms receipt; error handling is internal.
